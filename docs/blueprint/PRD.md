# 단일 서버 기반 토이 프로젝트 인프라 요구사항(요약)

## 1) 목표와 운영 철학

- Spring Boot + MySQL 기반 토이 프로젝트 여러 개를 “가용성/이중화는 포기”한 형태로 **한 대의 머신**에 몰아서 운영할 수 있어야 한다.
- 클라우드 벤더 락인 없이 실행 가능해야 하며, 이관이 쉽도록 docker compose를 활용해 서비스를 컨테이너 단위로 관리해야 한다.
- 각 서비스는 피크 기준 **최대 10 RPS**를 감당할 수 있어야 한다.
- 이 서비스는 외부 네트워크에 {hostname}으로 노출될 수 있어야 한다.

## 1-1) 용어정의

- 중앙 인프라: 본 인프라를 지칭한다.
- 서비스: 모든 Docker Compose Service를 뜻한다.
- 애플리케이션: 각 토이 프로젝트 애플리케이션을 뜻한다.
  - 각 애플리케이션은 현재는 Spring이지만, Express.js 등 중앙 인프라를 활용하고자 하는 임의의 서비스일 수 있다.
- 총 인프라: 중앙 인프라와 모든 애플리케이션을 묶어 뜻한다.

## 2) 중앙 인프라 요구사항

- 중앙 인프라는 다음 기능을 제공한다.
  - MySQL 서비스: 각 애플리케이션이 단일 MySQL 서비스에서 내부 database를 논리적으로 나누어 활용한다.
  - 메트릭/로그 모니터링 서비스: 애플리케이션, 호스트, 데이터베이스를 관제할 수 있어야 한다.
  - 정적 콘텐츠 서빙, HTTPS Termination 등을 지원해야 한다.
- 중앙 인프라의 네트워크 식별자는 다음 규칙을 따른다.
  - 인트라넷에서는 `central-` prefixed 호스트네임을 가진다. (ex: `central-mysql`)
  - 인터넷에서는 `central.hostname`으로 허용된 엔드포인트에 한해 접근 가능해야 한다.

## 3) 애플리케이션별 실행/분리 모델

- 각 애플리케이션은 인프라배포 (attach) 모드를 지원해야 한다.
  - **중앙 인프라에 연결**해서 운영하기 위한 세트이다.
- 각 애플리케이션은 원한다면 독자배포 (standalone) 모드를 지원할 수 있다.
  - 스스로 운영 가능한 세트이며, 중앙 인프라 없이 작동해야 한다.
  - 인프라배포와 독자배포에서 공유되는 설정은 쉽게 통합할 수 있어야 한다.
- 각 애플리케이션 자체는 인프라가 어떻게 구성되었는지에 불투명하게 작동해야 한다.
  - 애플리케이션이 인프라를 읽어오고 이해하는 것이 아니라, 인프라가 애플리케이션을 이해하고 감싸줘야 한다.
- 각 앱의 네트워크 식별자는 다음 규칙을 따른다.
  - 인트라넷에서는 `app-{appName}-` prefixed 호스트네임을 가진다. (ex: `app-myservice-server1`)
  - 인터넷에서는 `{appName}.app.hostname`으로 허용된 엔드포인트에 한해 접근 가능해야 한다.

## 3-1) 애플리케이션 수명주기

- 각 애플리케이션은 중앙 인프라 및 다른 애플리케이션의 중단 없이 새로 추가되거나 제거될 수 있어야 한다.
  - 어떤 방식으로 추가 및 제거를 지원할지는 고민이 필요하다.
- 총 인프라는 중단되지 않음을 상정하지만, 다른 머신으로 마이그레이션을 위해 중단되거나 시작할 수 있다.
  - 총 인프라가 중단되거나, 심지어는 마이그레이션되더라도 현재 운영중인 애플리케이션 세트 및 그들의 데이터, 모니터링 등은 모두 보존되어야 한다.
  - 애플리케이션이 추가된 이후 총 인프라가 중단 -> 다른 머신으로 마이그레이션 -> 총 인프라가 시작해도 애플리케이션을 다시 추가할 필요가 없어야 한다.
  - 마이그레이션을 편하게 할 수 있는 방법은 고민이 필요하다. (전체 .gzip 압축 후 이동?)

## 4) 성공 기준(완료 정의)
- 한 대의 머신에서 중앙 인프라 + 10개 서비스가 동시에 기동되고, 10대의 서비스를 더 추가한 뒤 15대를 삭제한 뒤 마이그레이션이 가능해야 한다.
- 서비스가 늘어나도(예: 10개 유지/확장) 실행 방식이 바뀌지 않고, 클라우드/온프레/다른 VM으로 옮겨도 Compose 조합만으로 동일하게 재현 가능해야 한다.
- 모니터링은 “각 서비스 상태와 기본 리소스 추세를 확인할 수 있는 수준”이면 충분하며, 고가용성·무중단 배포·멀티리전 등은 요구사항에서 제외한다.
